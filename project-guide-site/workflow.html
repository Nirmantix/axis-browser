The workflow: observe, act, prove.

Start with the rendered browser. Decide shared or isolated mode, capture real state, then choose whether the task needs quick diagnostics, audited evidence, or a rerunnable script.

Shared Chrome debugging

Use this for web apps, desktop apps with webviews, SaaS dashboards, internal tools, auth flows, and anything where real browser state matters.

# Point Axis Browser at the shared Chrome CDP endpoint.
export CHROME_DEVTOOLS_AXI_BROWSER_URL=http://127.0.0.1:9222
axis-browser stop
axis-browser pages
axis-browser open http://localhost:3000
axis-browser snapshot
axis-browser console --type error
axis-browser network --type fetch
Reproduce

Open the exact route or tab where the issue exists. Use the same login state if auth matters.

Inspect

Snapshot shows accessible structure and `@gN:N` refs. Console and network show runtime causes.

Interact

Click or fill refs exactly as printed. If stale, re-snapshot and retry.

Decide

Edit app code only after the browser evidence identifies a failing request, exception, or state mismatch.

Shared Chrome baseline

The standard shared-browser pattern is a dedicated Chrome profile exposing Chrome DevTools Protocol on port `9222`, with Axis Browser running its local bridge on port `9224`.

"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.axis-browser-data" \
  --no-first-run \
  --no-default-browser-check

export CHROME_DEVTOOLS_AXI_BROWSER_URL=http://127.0.0.1:9222
axis-browser stop
axis-browser pages
axis-browser snapshot
Dedicated profile

`~/.axis-browser-data` keeps automation login state separate from the everyday browser profile.

Bridge reset

`axis-browser stop` clears the persistent bridge so the next command uses the current environment and target.

Generation-tagged refs

Refs such as `@g1:3` must be passed back exactly as printed. `STALE_REF` means the page changed and a new snapshot is required.

Stop-guessing debug protocol

  1. 1
    Reproduce in the browser

    Open the target route or tab in the controlled shared browser session.

  2. 2
    Capture structure

    Run `axis-browser snapshot` and inspect the visible accessible state.

  3. 3
    Inspect runtime failures

    Run `axis-browser console` and `axis-browser network` before editing application code.

  4. 4
    Identify the concrete failure

    Look for the failing request, exception, disabled control, missing DOM state, or auth/session mismatch.

  5. 5
    Change code only after evidence

    Use browser observations to drive the implementation fix or the verified-run report.

Auth, tabs, and ports

Situation Operational pattern Command or note
Normal login state Launch the dedicated profile, log in manually once, then keep reusing the same `--user-data-dir`. ~/.axis-browser-data
SSO, MFA, or sensitive providers Log in without remote debugging, quit Chrome fully, then relaunch the same profile with `--remote-debugging-port=9222`. Use the same profile directory across both launches.
Tab source of truth Raw Chrome CDP is authoritative when tab selection looks wrong. curl -s http://127.0.0.1:9222/json/list
Chrome CDP port Default convention is `9222`. Change only when that port is already in use. CHROME_DEVTOOLS_AXI_BROWSER_URL=http://127.0.0.1:9333
Axis bridge port Default is `9224`. Change only when the local Axis bridge port conflicts. CHROME_DEVTOOLS_AXI_PORT=9225

Shared-browser troubleshooting

Bridge feels stale

  • reset `axis-browser stop`
  • check `axis-browser pages`
  • note Deep health checks recycle most stale bridge states automatically.

Bridge startup is slow

  • fast `npm install -g chrome-devtools-mcp`
  • timeout `CHROME_DEVTOOLS_AXI_BRIDGE_TIMEOUT_MS=60000`

Chrome CDP is unreachable

  • probe `curl -s http://127.0.0.1:9222/json/version`
  • cause Chrome was not launched with the expected debugging port or profile.

Login state is missing

  • profile Confirm the same `--user-data-dir` was reused.
  • sso For strict providers, log in without debugging first, then relaunch with debugging.

Verified run workflow

Use this when an operator, reviewer, or future agent needs checkable proof instead of a verbal claim.

# Scaffold a single-pass evidence bundle.
node "$SKILL_DIR/scripts/init-verified-run.mjs" \
  --task "Verify checkout renders an order confirmation" \
  --slug checkout-confirmation

# After filling plan.md and findings.md, validate it.
node "$SKILL_DIR/scripts/validate-verified-run.mjs" \
  --run .tmp/verified-runs/checkout-confirmation
Exit Status Operator action
0 pass All checks have valid PASS markers. Report the run and evidence.
1 fail:structure Fix malformed artifacts, missing markers, invalid IDs, or unsafe screenshot paths.
2 fail:checks Checks are structurally valid but failed or skipped. Report honestly or iterate.

Reusable workflow craft path

Use this when the deliverable is a browser workflow someone can rerun with different inputs.

Start craft mode

Craft mode changes the scaffold instructions. It does not magically generate automation.

node "$SKILL_DIR/scripts/init-verified-run.mjs" \
  --task "Search flights from SEA to JFK for supplied dates" \
  --slug flight-search \
  --mode craft
Prove once

Use Axis Browser, Browser Harness, or Playwright to complete the flow and collect evidence.

Refactor

Turn the working flow into `script.mjs` with explicit flags and defaults from `task.md`.

Prefer Playwright

For deterministic multi-page scripts, Playwright is the safer default backend.

Validate

Run the verified-run validator and report PASS, FAIL, or SKIP markers.
