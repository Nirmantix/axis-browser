Maintenance workflow.

Private operating guide for keeping the Axis Browser fork small, upgrade-friendly, and aligned with `kunchenguid/chrome-devtools-axi`.

When this workflow applies

Maintenance intent includes requests such as updating Axis Browser, syncing upstream, checking for upstream releases, bringing the repo current, or following the private maintenance playbook.

Primary intent

  • sync Check upstream live state before assuming an update exists.
  • inspect Review commits, tags, dependency changes, CI, docs, and release notes.
  • preserve Keep only intentional Axis Browser fork deltas.

Repository target

  • root Resolve with `git rev-parse --show-toplevel`.
  • remote `origin` remains `Nirmantix/axis-browser`.
  • upstream `upstream` remains `kunchenguid/chrome-devtools-axi`.

Private scope

  • private The private maintenance playbook stays local and untracked.
  • public Public docs stay focused on install, usage, and shared-Chrome workflow.

Repo guardrails

Area Current rule Reason
Branching `main` is the only long-lived branch. Use short-lived `codex/upstream-sync-YYYYMMDD` branches. Keeps the fork simple and protected.
Protected flow Open a PR into `main`, wait for `Kilo Code Review`, merge, delete the branch, then prune stale refs. Do not bypass protected `main` for small changes.
Release automation Release-please stays disabled unless explicitly re-enabled. Releases remain manual for this fork.
Nested skills `skills/browser-skill/` stays ignored by the parent repo and maintained as a standalone nested repo when present. The browser skill is not shipped through the parent Axis Browser release flow.

Non-negotiable fork invariants

Package identity

Keep package name `chrome-devtools-axi` for upstream compatibility.

Commands

Keep `axis-browser`, `axib`, and `chrome-devtools-axi` exposed.

Runtime state

Keep runtime state under `~/.axis-browser`.

Public wording

Public docs use `Axis Browser`, document GitHub-first install, and describe Windows support as partial.

Optional aliases

`axis`, `axi`, and `axisb` are not built-in commands unless locally created by an operator.

Small fork delta

Do not add unrelated product features to the fork.

Upstream sync workflow

  1. 1
    Confirm remotes and fetch

    Run `git remote -v`, `git fetch origin`, `git fetch upstream --tags`, and `git fetch --prune origin`.

  2. 2
    Check whether upstream changed

    Inspect `git log --oneline origin/main..upstream/main`, `git diff --stat origin/main..upstream/main`, upstream tags, and release notes.

  3. 3
    Create a short-lived branch

    Use a branch such as `codex/upstream-sync-YYYYMMDD` before merging upstream.

  4. 4
    Inspect high-risk files before merging

    Review `package.json`, `src/cli.ts`, `src/client.ts`, `src/bridge.ts`, `src/run.ts`, hooks, suggestions, tests, docs, and CI.

  5. 5
    Merge deliberately

    Use `git merge upstream/main`, resolve conflicts by preserving only needed Axis Browser deltas, and remove obsolete fork workarounds when upstream supersedes them.

  6. 6
    Re-assert invariants

    Verify package name, bins, branding, state directory, build behavior, docs, changelog, and platform-support wording.

  7. 7
    Validate and PR

    Run the appropriate tests and build, inspect the public surface, push the branch, open a PR, wait for required review, then clean up.

Public documentation rules

README.md

Public product positioning, install/use guidance, compatibility notes, platform support, and GitHub-first install path.

Shared workflow guide

`docs/vibe-coding-browser-workflow.md` owns `9222` usage, bridge reset habits, tab troubleshooting, auth/session guidance, and local-helper caveats.

CHANGELOG.md

Keep inherited upstream history intact and keep the Axis Browser `Unreleased` section current when public behavior changes.

Never publish private process

Do not track the private maintenance playbook, private path prompts, Airlock scaffolding, or public docs whose main purpose is private fork maintenance.
