Two setup routes: current workstation or fresh machine.

The browser-skill folder is portable. Installed browser tools are machine-local. Each target project still needs `.tmp/` artifact setup, and Playwright stays project-local for deterministic scripts, traces, or CI.

Portability boundaries

Thing Portable by copying or pointing at `skills/browser-skill/`? Still needed per project or machine?
Workflow brain Yes. The skill router, references, and scripts can be reused from this checkout. The agent must know the path through `BROWSER_SKILL_DIR`, `AXIS_BROWSER_HOME`, or an agent-supported skill location.
Axis Browser CLI No. The skill can call it only if the CLI is installed on the machine. Install globally once per machine, or run from a local checkout.
Browser Harness, optional CLIs, Playwright browsers No. These are machine/project dependencies, not bundled inside the skill. Use `check-prerequisites.sh` to see what is available; install Playwright inside target projects only when needed.
Project artifacts No. `.tmp/` belongs to the target project. Run `setup.sh` once inside every project where browser evidence will be saved.

Route A: current macOS workstation

# Run from any target project on this workstation.
export AXIS_BROWSER_HOME="/path/to/axis-browser"
export BROWSER_SKILL_DIR="$AXIS_BROWSER_HOME/skills/browser-skill"
export SKILL_DIR="$BROWSER_SKILL_DIR"

# Preview the only project-local writes.
bash "$SKILL_DIR/scripts/setup.sh" --dry-run

# Create .tmp/ artifact folders and .gitignore hygiene in that project.
bash "$SKILL_DIR/scripts/setup.sh"

# Confirm which browser powers are actually available in that project.
bash "$SKILL_DIR/scripts/check-prerequisites.sh"
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --print-install-commands

For local projects on this workstation, a central skill path is usually cleaner than copying the skill into every repository. Project-local copies are better when a repository must pin and carry its own workflow version.

Best for personal projects

Use the central checkout path above. Updating this one skill updates the workflow available to all local projects.

Best for team repos

Copy the skill to `.agents/skills/browser-skill/` or the host-specific project path when the repo should carry the workflow with it.

Still project-local

Install Playwright in the app repo when the task needs reusable scripts, traces, network interception, visual regression, or CI.

Route B: fresh machine

# 1. Install global runtime prerequisites.
node --version
npm --version

# 2. Install Axis Browser globally from this fork when missing.
npm install -g github:Nirmantix/axis-browser

# Or with Bun when Bun is the selected global package manager.
bun add -g github:Nirmantix/axis-browser

axis-browser --version
axib --version
chrome-devtools-axi --version

# 3. Install browser-skill itself somewhere the agent can find it.
mkdir -p "$HOME/.config/agents/skills"

# Copy browser-skill from a trusted checkout, shared archive, or internal repo.
cp -R /path/to/browser-skill "$HOME/.config/agents/skills/browser-skill"

export BROWSER_SKILL_DIR="$HOME/.config/agents/skills/browser-skill"
export SKILL_DIR="$BROWSER_SKILL_DIR"

# 4. Review and install machine-level browser workflow tools.
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --install

# 5. In each target project, initialize artifacts and verify tools.
cd /path/to/target-project
bash "$SKILL_DIR/scripts/setup.sh"
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --print-install-commands

`npx -y chrome-devtools-axi` resolves the upstream npm package, not the Axis Browser GitHub fork. Use the GitHub install path for this fork.

Skill resolution order for prompt-driven setup: existing `BROWSER_SKILL_DIR`, repository-local `./skills/browser-skill`, `$AXIS_BROWSER_HOME/skills/browser-skill`, then standard agent skill locations.

macOS

Use Homebrew or standard installers for Node.js, `uv`, `jq`, ImageMagick, and optional CLIs. Axis Browser itself is a Node global install.

Linux

The core CLI and browser-skill work. Use the distro package manager for Node.js 20+, Chromium/Chrome, `uv`, and optional utilities.

Windows

Core CLI support is partial today. Prefer WSL2 or macOS/Linux for the documented shared-Chrome shell workflow until Windows helpers mature.

Verify instead of guessing

bash "$SKILL_DIR/scripts/check-prerequisites.sh"
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --print-install-commands
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --install --dry-run
bash "$SKILL_DIR/scripts/check-prerequisites.sh" --update --dry-run
axis-browser open https://example.com
axis-browser snapshot

Default and print modes are read-only. Install and update modes are permission-gated, keep credentials out of files, and never install project-local libraries globally.
